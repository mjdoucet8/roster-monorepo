# specs

Use this folder for specifications: clear descriptions of what should be built, changed, or supported.

A good spec explains the desired behavior, user or system needs, constraints, non-goals, edge cases, and acceptance criteria. It should be detailed enough that someone could implement the work later without needing the original conversation.

Create a spec when the work has meaningful product behavior, architecture impact, integration details, or enough ambiguity that jumping straight into code would lose important context.
